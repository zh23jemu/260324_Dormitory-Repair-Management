## 摘要

随着高校后勤信息化建设的不断推进，学生宿舍日常维修服务逐渐由传统线下登记方式转向线上统一管理。传统宿舍报修流程普遍存在登记分散、信息传递效率低、维修进度不透明、责任界定不清、统计分析困难等问题，难以满足高校后勤管理精细化、可追溯和高效率的实际需求。针对上述问题，本文结合高校宿舍报修业务场景，设计并实现了一套基于 Spring Boot 与 Vue 的高校宿舍报修管理系统。该系统围绕学生报修、宿管审核分配、维修人员接单反馈、学生评价和管理员统计分析等核心业务流程展开，形成了较为完整的宿舍报修闭环。

本系统采用前后端分离架构，后端使用 Spring Boot 作为基础开发框架，结合 Spring Security 与 JWT 实现用户认证和角色权限控制，使用 SQLite 作为轻量级关系数据库完成数据存储，并通过本地文件存储机制实现报修图片和维修结果图片的上传与访问。前端部分按照角色使用场景划分为两个终端：一是面向宿管、维修人员和管理员的 PC 管理端，采用 Vue 3、Vite、Element Plus 和 ECharts 实现数据管理与统计可视化；二是面向学生的移动 Web 端，采用 Vue 3、Vite 和 Vant 构建轻量化交互界面，支持在手机浏览器中完成登录、注册、提交报修、查看进度和评价反馈等操作。

在系统设计过程中，本文围绕业务需求分析、功能模块划分、数据库结构设计、接口设计、权限控制设计和工单状态流转设计等方面进行了详细论述。系统共设置学生、宿管人员、维修人员和管理员四类角色，并依据不同角色职责划分功能边界。学生端实现了注册登录、公告查看、报修提交、图片上传、工单查询和服务评价等功能；宿管端实现了工单审核、驳回、分配维修人员、楼栋和宿舍信息管理、公告管理等功能；维修人员端实现了工单接单、维修反馈、结果图片上传和个人工作统计；管理员端实现了用户管理、系统日志查看以及报修类型、楼栋热度、满意度等统计分析功能。

为验证系统设计的合理性与实现的可用性，本文结合项目运行环境与示例数据，对登录认证、工单流转、报修反馈、图片上传、公告管理和统计分析等关键业务环节进行了功能测试和结果分析。测试结果表明，该系统能够较好地满足高校宿舍报修管理的基本业务需求，具有流程清晰、结构简洁、部署方便和演示效果直观等特点，适合作为高校宿舍报修场景下的信息化管理原型系统。

本文的研究与实现工作对高校宿舍报修业务的信息化、标准化和流程化管理具有一定参考价值。同时，系统仍存在消息提醒、自动派单、权限粒度细化和移动端功能扩展等方面的改进空间，后续可在此基础上进一步完善和推广应用。

## Abstract

With the continuous development of informatization in university logistics management, dormitory maintenance services are gradually shifting from traditional offline registration to unified online management. Conventional repair processes often suffer from scattered records, low information transmission efficiency, opaque maintenance progress, unclear responsibility tracing, and weak statistical analysis capability. To address these problems, this paper designs and implements a university dormitory repair management system based on Spring Boot and Vue.

The system adopts a front-end and back-end separated architecture. The back end is built with Spring Boot, while Spring Security and JWT are used to implement authentication and role-based access control. SQLite is selected as the database because of its lightweight deployment characteristics, and local file storage is used to support image upload and access. According to different user scenarios, the front end is divided into two clients: a PC management client for dormitory administrators, maintenance staff and system administrators, and a mobile Web client for students. The PC client is developed with Vue 3, Vite, Element Plus and ECharts, while the mobile client is implemented with Vue 3, Vite and Vant.

This paper discusses the requirement analysis, functional module division, database design, interface design, permission control and repair order state transition in detail. The system defines four roles, including student, dormitory administrator, maintenance staff and administrator. Students can register, log in, submit repair requests, upload fault images, check repair progress and submit ratings. Dormitory administrators can review and reject repair orders, assign repair staff, manage dormitory information and publish announcements. Maintenance staff can accept assigned orders, submit maintenance results and upload result images. Administrators can manage users, view system logs and analyze statistics such as repair type distribution, building repair heat and rating results.

Functional testing and result analysis are carried out for key business links such as authentication, repair order circulation, image upload, announcement management and statistical analysis. The test results show that the system can meet the basic requirements of dormitory repair management in universities. It has the advantages of a clear workflow, simple structure, convenient deployment and intuitive demonstration effect. Therefore, it can serve as a practical prototype for the informatization of university dormitory maintenance management.

Although the system has completed the main business loop, there is still room for further improvement in message notification, automatic dispatching, finer-grained permission management and richer mobile-side functions. Future work can focus on these aspects to improve the practicality and scalability of the system.

**Key Words:** dormitory repair; Spring Boot; Vue 3; SQLite; front-end and back-end separation; maintenance management

# 1 绪论

## 1.1 研究背景

随着高校办学规模不断扩大，学生宿舍数量持续增加，宿舍基础设施的维修工作逐渐成为校园后勤管理中的高频事务。学生在日常住宿过程中经常会遇到照明设备损坏、水龙头漏水、门锁卡顿、网络异常、家具松动、热水器故障等问题。这类问题若不能被及时发现和处理，轻则影响学生的学习与生活体验，重则可能带来安全隐患。传统的宿舍报修方式通常依赖纸质登记、电话报修或者通过宿管口头转达，其主要问题在于登记口径不统一、信息流转过程较长、责任主体不明确、工单进度难跟踪、统计分析难开展。

在高校信息化建设不断深入的背景下，将学生宿舍报修管理流程迁移到信息系统中已经成为提升后勤服务水平的重要手段。通过建立规范化、流程化、可追溯的宿舍报修管理系统，不仅可以提升报修受理效率，缩短信息传递链路，还可以对宿舍维修数据进行统一管理和分析，为后勤部门进行资源配置、服务改进和日常决策提供数据支持。

当前，许多高校已开始在教务、图书、门禁、后勤等领域开展信息化系统建设，但针对学生宿舍报修这一细分业务场景的系统往往存在功能过于简单、移动端支持不足、流程闭环不完整、统计分析薄弱等问题。基于此，设计并实现一套贴合高校宿舍报修实际场景、功能完整且技术实现可控的毕业设计系统，具有较强的实践意义。

## 1.2 研究目的与意义

本文研究的主要目的是设计并实现一套高校宿舍报修管理系统，使学生、宿管人员、维修人员和管理员能够围绕报修业务形成完整的协同处理机制。通过该系统，学生可以方便地在线提交报修申请并实时查看处理进度；宿管人员可以对报修信息进行审核和分配；维修人员可以接收任务、反馈结果并上传维修图片；管理员则可以对用户、日志和统计数据进行统一管理。

从理论意义来看，本文将高校宿舍报修业务与 Web 信息系统开发技术相结合，对多角色流程型业务系统的需求分析、架构设计、数据库设计和状态流转设计进行了系统梳理，体现了信息系统分析与设计课程相关知识在具体应用场景中的落地过程。

从实践意义来看，该系统有助于改善传统报修模式下的信息断层问题。系统通过统一工单编号、状态流转记录和日志记录，使报修流程更加清晰可追踪；通过图片上传和维修结果反馈，提高信息表达的完整性；通过管理员端的统计分析和可视化展示，为宿舍设施维护工作提供直观的数据依据；通过学生端移动 Web 页面，降低使用门槛，提高学生报修参与度。

此外，本文所实现的系统采用 Spring Boot、Vue 3 和 SQLite 等较为成熟且适合教学和毕业设计场景的技术方案，部署成本低、运行方式清晰、功能演示直观，可作为同类校园后勤管理系统开发的参考原型。

## 1.3 国内外研究现状

从国内研究情况来看，随着高校信息化建设和智慧校园概念的发展，后勤维修、资产管理、学生服务等系统受到越来越多关注。王剑围绕三明学院学生宿舍报修管理业务开展了研究，对学生宿舍报修流程和管理痛点进行了分析，为本文选题提供了较直接的业务背景参考[1]。李文杰在基于 Spring Boot 与 Vue 的系统研发中指出，前后端分离架构在提升系统开发效率、增强界面交互体验和降低维护耦合度方面具有明显优势[2]。这类研究表明，采用主流 Web 技术栈构建中小型业务信息系统已经成为较成熟的实现路径。

在数据库和系统分析方面，《数据库系统概论》对关系数据库设计方法、实体联系建模和数据一致性控制进行了系统阐述[3]；Java Web 相关教材对典型 Web 系统的分层结构、请求响应处理和服务端业务组织方式进行了说明[4]；Spring Boot 开发教程则从工程搭建、配置管理、接口开发和安全控制等角度提供了完整实践基础[5]。前端方面，Vue 技术相关著作强调了组件化开发、响应式数据管理以及视图驱动开发模式对提升复杂页面开发效率的重要作用[6]。

从国外同类研究的总体趋势来看，高校及公共设施维修管理系统更加注重流程可视化、移动端适配、自动派单、消息通知与数据分析能力。一些大型维修管理平台会引入更复杂的工单优先级、服务级别协议、设备资产编码和多终端协作机制。但对于毕业设计项目而言，过于复杂的企业级能力并不适合直接照搬。相比之下，围绕明确业务流程完成“提交、审核、分配、处理、反馈、评价、统计”的核心闭环，更符合高校毕业设计的规模和实现边界。

综合已有研究可以看出，高校宿舍报修管理系统的开发重点并不在于堆叠复杂技术，而在于围绕业务流程进行合理建模、准确划分角色职责、保证数据一致性并提供良好的交互体验。因此，本文选择以“功能闭环完整、实现难度可控、便于演示与答辩”为设计原则，构建一套适合本科毕业设计场景的高校宿舍报修管理系统。

# 2 相关技术与理论基础

## 2.1 Spring Boot 技术

Spring Boot 是当前 Java Web 开发中应用广泛的后端框架之一，其核心优势在于能够通过约定优于配置的方式快速搭建独立运行的应用程序。相较于传统 Spring 项目需要手动配置大量 XML 或 Java 配置，Spring Boot 通过自动装配机制显著降低了项目初始化和模块集成成本，适合用于开发中小型业务系统[5]。

在本系统中，Spring Boot 主要承担以下职责：一是提供 Web 接口开发能力，实现登录、报修、分配、反馈、统计等业务接口；二是结合 Spring Security 完成接口访问控制和认证过滤；三是通过配置文件统一管理服务端口、数据库连接、上传目录等运行参数；四是借助依赖注入和分层结构组织系统代码，使 Controller、Service、配置类和公共组件之间职责清晰。

采用 Spring Boot 的原因主要有三点。首先，任务书对技术路线提出了 Spring Boot 与 Vue 的要求；其次，Spring Boot 生态成熟，与安全认证、数据库访问、文件上传等功能集成便利；最后，Spring Boot 项目结构清晰，便于在毕业设计论文中展开系统分析与实现说明。

## 2.2 Vue 3 与前端组件化开发

Vue 3 是一种轻量级、渐进式的前端框架，具有响应式数据管理、组件化开发和模板语法简洁等特点[6]。在本系统中，Vue 3 被用于实现前端页面逻辑，通过状态绑定、事件响应和异步接口调用完成页面与后端数据的联动。

本系统前端按照不同使用群体划分为两个独立应用。PC 管理端采用 Vue 3、Vite 和 Element Plus，适合实现工单列表、表格展示、对话框、表单录入和图表看板等后台场景；学生移动端采用 Vue 3、Vite 和 Vant，侧重手机浏览器中的轻量操作体验，便于学生快速完成登录、报修和查询。

Vue 3 的组件化开发思想使页面逻辑可以围绕业务状态组织。例如，管理端通过标签页划分工单管理、宿舍管理、公告管理、统计分析等区域；学生端通过登录状态切换登录注册页和业务页。前端统一使用 Axios 向后端接口发起请求，并在请求拦截器中自动附加 JWT Token，从而简化权限认证流程。

## 2.3 SQLite 数据库

SQLite 是一种轻量级嵌入式关系数据库，具有部署简单、无需单独安装数据库服务、占用资源少和适合中小规模应用等特点[3]。与 MySQL、PostgreSQL 等传统独立数据库相比，SQLite 更适合毕业设计项目和原型系统，能够降低环境搭建难度，提高项目的可移植性和演示便利性。

本系统使用 SQLite 存储用户信息、学生档案、楼栋宿舍、报修工单、工单图片、流转记录、维修结果、评价信息、公告、日志和字典配置等数据。数据库文件以 `dorm_repair.db` 形式位于项目根目录。项目在服务启动时根据 `schema.sql` 自动建表，并结合 `data.sql` 导入示例数据，从而保证系统在首次运行时即可完成业务演示。

选择 SQLite 的主要原因在于：第一，本系统属于中小型单体应用，数据量和并发规模有限；第二，毕业设计更关注业务流程的完整性，而非复杂数据库集群能力；第三，SQLite 的文件型数据库特征便于项目打包、迁移与答辩演示。

## 2.4 JWT 与权限控制机制

JWT 即 JSON Web Token，是一种基于 Token 的身份认证方案。服务端在用户登录成功后生成携带用户身份信息和角色信息的令牌，客户端在后续请求中通过请求头携带该令牌，服务端再解析并验证其有效性。与传统基于 Session 的认证方式相比，JWT 更适合前后端分离系统，因为其能够实现无状态认证，减少服务端会话维护开销。

本系统中，用户登录接口返回 Token，前端分别将管理员端 Token 和学生端 Token 保存到浏览器本地存储中。后端通过 `JwtAuthenticationFilter` 对进入系统的请求进行统一拦截和解析，并将登录用户信息写入安全上下文。`SecurityConfig` 中配置了 `/api/auth/**`、`/api/common/upload` 和 `/uploads/**` 等公开访问路径，其他接口均要求认证后访问。

在角色权限方面，系统采用基于角色字符串的简单权限控制方式。业务层通过 `SecurityUtils.requireRole` 对访问者角色进行校验，例如学生接口仅允许 `student` 角色访问，宿管相关接口允许 `dorm_admin` 和 `admin` 角色访问，管理员接口只允许 `admin` 访问。该方式虽然不具备复杂 RBAC 的灵活性，但足以满足当前项目的业务边界和实现复杂度要求。

## 2.5 RESTful API 与前后端分离模式

前后端分离是当前 Web 系统开发中的常见模式，其核心思想是将后端业务服务与前端界面渲染解耦。后端以接口形式提供数据和业务能力，前端负责交互逻辑和界面展示，二者通过 HTTP 协议交换 JSON 数据。

本系统所有业务接口统一以 `/api` 为前缀，采用近似 RESTful 的方式组织资源路径。例如，认证相关接口位于 `/api/auth`，学生端业务接口位于 `/api/student`，宿管端接口位于 `/api/dorm-admin`，维修人员接口位于 `/api/repairer`，管理员接口位于 `/api/admin`。接口返回采用统一结构，包含状态码、提示信息和数据内容，便于前端统一处理。

这种设计方式有利于系统扩展与维护。一方面，前端与后端职责划分清晰，可以独立开发和调试；另一方面，相同的后端接口可以被不同终端复用。本系统中的学生移动端和 PC 管理端虽然界面形态不同，但都调用同一套后端服务，从而减少重复开发工作。

# 3 需求分析

## 3.1 业务背景分析

宿舍维修是高校后勤服务中的基础业务之一，其管理质量直接影响学生的生活体验和学校后勤服务评价。在传统管理模式下，学生往往通过电话、纸质登记、微信群或面对面反馈方式提交报修需求。宿管人员接到反馈后，再通过电话或线下联系维修人员进行处理。此过程中存在以下问题：一是报修信息记录不规范，可能出现描述不完整、重复登记或遗漏；二是维修流程透明度低，学生无法及时了解进度；三是宿管与维修人员之间的任务交接依赖人工沟通，效率较低；四是缺乏统一的数据沉淀机制，不利于后续统计分析和后勤优化。

因此，建设一套面向宿舍维修场景的信息管理系统，有助于实现报修业务的标准化和流程化。系统应能够在“学生提交报修、宿管审核分配、维修人员处理反馈、学生评价完成”的业务链条上形成完整闭环，并且具备基础的公告发布、宿舍信息维护、用户管理和数据统计能力。

## 3.2 角色需求分析

### 3.2.1 学生角色需求

学生是系统中的报修发起者，其主要需求包括：

1. 能够完成注册和登录，建立个人身份信息。
2. 能够在移动端便捷地提交报修申请。
3. 能够填写报修标题、故障描述、期望维修时间，并上传现场图片。
4. 能够查看个人报修列表和详细处理进度。
5. 能够在维修完成后查看结果并进行评价。
6. 能够查看宿舍相关公告信息。

### 3.2.2 宿管人员需求

宿管人员是报修业务中的关键调度角色，其主要需求包括：

1. 能够登录管理端并查看待审核工单。
2. 能够审核学生提交的报修申请。
3. 对符合条件的工单进行维修人员分配，对信息不完整的工单进行驳回。
4. 能够查看工单详情、流转记录和图片信息。
5. 能够维护楼栋、宿舍以及相关公告信息。
6. 能够导出部分业务数据，方便日常管理。

### 3.2.3 维修人员需求

维修人员承担工单执行职责，其主要需求包括：

1. 能够登录系统查看分配给自己的工单。
2. 能够对待接单工单执行接单操作。
3. 能够提交维修结果、维修材料和完成时间。
4. 能够上传维修后的结果图片。
5. 能够查看个人工单统计情况。

### 3.2.4 管理员需求

管理员负责系统层面的综合管理，其主要需求包括：

1. 能够管理系统用户信息。
2. 能够修改用户状态和重置密码。
3. 能够查看系统日志和统计分析结果。
4. 能够查看报修类型分布、楼栋报修热度和满意度数据。
5. 能够查看字典配置等基础数据。

## 3.3 功能需求分析

根据角色职责划分，系统应至少实现以下功能模块。

学生端功能模块包括：登录注册、查看公告、提交报修、上传故障图片、查看工单列表、查看工单详情、提交评价、查看个人信息。

宿管端功能模块包括：工单审核、工单驳回、维修人员分配、查看工单详情、楼栋管理、宿舍管理、学生住宿信息调整、公告管理。

维修人员端功能模块包括：查看我的工单、接单、提交维修反馈、上传维修结果图片、查看个人统计。

管理员端功能模块包括：用户管理、用户状态更新、密码重置、统计分析、日志查看、字典查看。

此外，为了支撑核心业务，还需要实现通用上传接口、统一认证机制、统一异常处理以及系统初始化数据导入能力。

## 3.4 非功能需求分析

### 3.4.1 易用性需求

系统用户包括学生、宿管人员、维修人员和管理员，不同用户的计算机水平和使用习惯存在差异，因此系统界面应尽可能简洁直观。学生端应适配手机浏览器访问，减少复杂操作步骤；管理端应以表格、表单和图表为主，便于快速查看与处理业务信息。

### 3.4.2 安全性需求

系统需要对不同角色进行身份认证和权限控制，防止未登录访问和越权操作。对于用户信息、工单信息和图片资源，需要建立合理的访问边界。虽然当前版本未引入复杂的密码加密和审计体系，但至少应保证基础的登录校验、Token 校验和角色限制机制。

### 3.4.3 可维护性需求

系统应采用清晰的分层结构，使接口层、业务层、配置层和前端页面逻辑彼此分离，便于后续维护与扩展。数据库设计应围绕核心业务实体展开，避免无序扩张。接口路径、返回结构和状态值命名应尽量统一。

### 3.4.4 可靠性需求

宿舍报修业务具有一定连续性，系统应能够保证关键数据不丢失。报修工单状态变更需要有明确记录，上传文件需要保证路径可访问，异常情况应能够返回明确提示。

### 3.4.5 性能需求

由于本系统定位于中小型单体应用原型，主要服务于毕业设计演示和一般规模业务验证，因此无需追求高并发架构。但系统应在常规开发环境下稳定运行，能够完成多角色访问、列表查询、图片上传和统计展示等基础操作。

## 3.5 业务流程分析

### 3.5.1 学生报修流程

学生登录系统后，在移动端填写报修标题、故障描述、期望维修时间和报修类型，并上传现场图片。系统根据学生档案中的楼栋和宿舍信息生成报修工单，初始状态为“待审核”。此后学生可以在个人工单列表中查看工单状态，并在维修完成后进行评分和反馈。

### 3.5.2 宿管审核分配流程

宿管人员登录 PC 管理端后，查看待审核工单。若报修信息有效且可以安排维修，则宿管为该工单分配维修人员，工单状态由“待审核”变为“待接单”；若信息不完整或不符合处理条件，则填写驳回原因，工单状态变为“已驳回”。

### 3.5.3 维修处理流程

维修人员查看分配给自己的工单，对“待接单”工单执行接单操作，使工单状态变为“处理中”。维修完成后，维修人员填写维修结果、使用材料、完成时间，并上传维修结果图片。此时工单状态变为“待评价”。

### 3.5.4 学生评价流程

当工单进入“待评价”状态后，学生在移动端打开对应工单详情，查看维修结果和图片后提交评分与反馈内容。系统保存评价记录并将工单状态更新为“已完成”，至此整个报修业务闭环完成。

### 3.5.5 管理统计流程

管理员登录后可以查看全局统计数据，包括总工单数、待审核工单数、处理中工单数、已完成工单数、平均评分、报修类型分布、楼栋报修热度和满意度统计。通过这些数据，管理员可以更直观地掌握系统运行情况和宿舍维修服务情况。

# 4 系统总体设计

## 4.1 设计目标与原则

本系统在设计时遵循以下原则。

第一，功能闭环完整。系统应至少覆盖宿舍报修场景中的核心业务链路，确保从报修申请到评价完成的流程能够贯通。

第二，实现复杂度可控。作为本科毕业设计项目，系统不追求企业级复杂度，而是在保证功能完整的前提下，采用适合教学和答辩演示的技术方案。

第三，界面分工明确。根据使用场景划分 PC 管理端与移动端学生端，使不同角色在适合自身场景的界面中完成业务操作。

第四，数据结构清晰。围绕用户、宿舍、工单、流转、反馈和评价等核心实体完成数据库设计，保证数据之间的关系明确可追溯。

第五，部署与演示方便。系统采用 SQLite 和本地文件上传路径，降低环境配置难度，便于快速启动和演示。

## 4.2 系统架构设计

本系统采用典型的前后端分离三层结构，由移动端学生前端、PC 管理端前端和 Spring Boot 后端三部分组成。

前端层中，学生端主要运行在手机浏览器中，负责接收学生输入、提交报修和展示工单详情；管理端主要运行在 PC 浏览器中，负责宿管、维修人员和管理员的业务操作与数据管理。

业务服务层由 Spring Boot 提供，负责处理登录认证、权限校验、工单流转、宿舍管理、公告管理、统计分析和文件上传等业务逻辑。系统按照 Controller、Service、Config、Common 等结构组织代码，便于职责划分。

数据存储层由 SQLite 数据库与本地上传目录组成。结构化业务数据存储在 `dorm_repair.db` 中，图片文件存储在 `uploads` 目录中，并通过静态资源映射向外提供访问路径。

从请求流程来看，前端通过 Axios 调用后端 REST 接口；后端在认证成功后将业务结果以统一 JSON 结构返回前端；前端再根据结果更新界面状态。该结构既保证了系统模块边界清晰，也便于后续增加新终端或扩展接口。

## 4.3 功能模块设计

### 4.3.1 学生端模块

学生端包含登录注册模块、公告查看模块、报修提交模块、工单查询模块、工单详情查看模块和评价反馈模块。其核心目标是让学生能够在移动端完成报修全流程中的发起、查询与评价环节。

### 4.3.2 宿管端模块

宿管端主要包括工单管理模块、楼栋与宿舍管理模块、学生住宿信息调整模块和公告管理模块。其中工单管理模块是宿管端的核心，承担审核、分配和驳回职责。

### 4.3.3 维修人员模块

维修人员模块包括我的工单、工单详情、接单处理、维修结果反馈和个人统计。该模块的重点在于将维修执行过程结构化记录下来。

### 4.3.4 管理员模块

管理员模块包括用户管理、日志管理、统计分析和字典查看。管理员作为系统的最高权限角色，主要承担全局管理与查看职责。

## 4.4 数据库设计

本系统数据库围绕报修业务流程设计，共包含以下核心数据表：

`user` 用户表：保存账号、密码、姓名、电话、角色、状态等基础信息。  
`student_profile` 学生档案表：保存学号、学院、专业、班级以及宿舍绑定信息。  
`dorm_building` 楼栋表：保存楼栋名称、楼栋编码和层数。  
`dorm_room` 宿舍表：保存楼栋关联、宿舍号、容量和设施信息。  
`repair_type` 报修类型表：保存水电、家具、门窗、网络等类别。  
`repair_order` 工单表：保存报修业务核心数据，是系统最重要的业务表。  
`repair_order_image` 工单图片表：保存故障图片和维修结果图片的路径。  
`repair_flow` 工单流转表：记录工单状态变化过程。  
`repair_feedback` 维修反馈表：保存维修结果、材料和完成时间。  
`repair_rating` 评价表：保存学生评分与评价内容。  
`announcement` 公告表：保存系统公告信息。  
`sys_log` 系统日志表：保存关键操作记录。  
`sys_dict` 字典表：保存基础字典配置。

从实体关系上看，用户与学生档案之间是一对一关系；楼栋与宿舍是一对多关系；工单与工单图片、流转记录之间是一对多关系；工单与维修反馈、评价之间是一对一关系。这样的设计既能满足业务表达需要，也便于通过 SQL 进行详情查询和统计分析。

## 4.5 接口设计

系统接口统一以 `/api` 为前缀，并按角色和业务类型分类组织。

认证接口包括登录、注册、获取当前用户、修改密码和重置密码；学生端接口包括公告获取、报修提交、工单列表、工单详情、评价提交和个人信息维护；宿管端接口包括工单列表、工单详情、分配、驳回、楼栋管理、宿舍管理、学生住宿信息调整和公告管理；维修人员接口包括我的工单、工单详情、接单、维修反馈和个人统计；管理员接口包括用户管理、统计接口、日志接口和字典接口；通用接口包括文件上传接口。

接口返回采用统一结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

统一的接口风格有助于前端进行统一错误处理和数据解析，也便于论文中对接口体系进行系统化说明。

## 4.6 工单状态流转设计

工单状态设计是本系统的核心之一。系统定义了六种状态：

`pending_review`：待审核  
`rejected`：已驳回  
`pending_accept`：待接单  
`processing`：处理中  
`pending_rating`：待评价  
`completed`：已完成

其状态流转逻辑如下：学生提交报修后生成待审核工单；宿管驳回则进入已驳回状态；宿管分配维修人员后进入待接单状态；维修人员接单后进入处理中状态；维修完成后进入待评价状态；学生评价后进入已完成状态。

为了保证业务过程可追踪，系统在每次关键状态变化时都会向 `repair_flow` 表插入一条流转记录。这样既便于后续详情页展示，也为管理员和答辩演示提供了完整的数据支撑。

# 5 系统详细设计与实现

## 5.1 后端总体实现

后端工程位于 `server` 目录，采用 Spring Boot 构建。主要目录包括 `controller`、`service`、`config`、`security`、`dto`、`common` 和 `util` 等。接口控制器负责接收请求和返回统一响应，业务层负责核心逻辑处理，配置层负责安全配置和静态资源映射，公共模块负责统一返回结果、全局异常处理和业务异常定义。

系统共实现 `AuthController`、`StudentController`、`DormAdminController`、`RepairerController`、`AdminController`、`CommonController` 和 `UploadResourceController` 七个控制器，基本覆盖了认证、学生业务、宿管业务、维修业务、管理员业务、上传能力和上传文件访问路径等所有主要功能。

与一些采用 ORM 框架进行复杂实体映射的系统不同，本项目以 `JdbcTemplate` 和 SQL 为主组织数据访问逻辑，配合统一的 `CommonQueryService` 完成列表和详情查询。这种实现方式虽然相对直接，但更有利于毕业设计中说明业务数据表与 SQL 语句之间的对应关系。

## 5.2 认证与权限模块实现

### 5.2.1 登录与注册实现

认证模块由 `AuthController` 和 `AuthService` 组成。登录时，系统根据用户名查询用户信息并校验密码，校验成功后生成 JWT Token 返回前端。前端在本地存储中保存 Token，并在后续请求中自动放入 `Authorization` 请求头。

注册功能面向学生开放。系统在注册时会检查用户名是否重复、学号是否重复，然后向 `user` 表和 `student_profile` 表同时写入数据。这样既保证了用户登录身份的建立，又同步创建了学生档案信息。

### 5.2.2 当前用户获取与密码修改

系统提供获取当前登录用户信息的接口，前端在登录成功后调用该接口以确定角色信息和初始化页面。系统还提供修改密码和管理员重置密码功能，用于满足基础账号维护需求。

### 5.2.3 安全过滤与角色控制

`SecurityConfig` 中关闭了传统表单登录和 Session 模式，启用了基于 JWT 的无状态认证。配置文件中明确放行认证相关接口、上传接口和上传资源访问路径，其余接口都需要认证后访问。

角色控制主要在业务层实现。例如学生相关服务调用 `SecurityUtils.requireRole("student")`，宿管相关服务调用 `SecurityUtils.requireRole("dorm_admin", "admin")`，管理员服务调用 `SecurityUtils.requireRole("admin")`。这种控制方式简单明确，能够在当前系统规模下满足不同角色的访问隔离要求。

## 5.3 学生端功能实现

学生端位于 `student-web` 目录，其页面入口为 `App.vue`。系统根据是否存在 `student_token` 判断当前用户是否已登录，未登录时显示登录与注册页面，已登录后显示业务页面。

### 5.3.1 公告查看实现

学生登录成功后，页面会调用 `/api/student/announcements` 接口获取公告列表，并在首页通过通知栏展示最新公告内容。该功能使学生能够及时看到宿舍维修相关通知和系统使用提示。

### 5.3.2 报修提交实现

学生在报修页面中填写标题、故障描述、期望时间和报修类型，并可以通过 `van-uploader` 组件选择最多三张图片。图片上传通过 `/api/common/upload` 完成，后端返回文件访问路径后，前端将路径加入 `repairForm.imagePaths`。当学生点击提交报修时，系统将表单数据发送至 `/api/student/repair-orders` 接口，后端在业务层中根据当前学生账号查出其绑定的楼栋和宿舍信息，生成带有工单编号的报修工单，并写入初始流转记录。

### 5.3.3 工单列表与详情实现

学生可以查看个人工单列表，每条工单显示标题、报修类型、楼栋宿舍和当前状态。点击具体工单后，系统会请求详情接口，返回工单基础信息、图片记录、流转记录和评价结果等内容。前端通过弹出层展示详情，支持查看维修人员姓名、驳回原因、维修结果等信息。

### 5.3.4 评价反馈实现

当工单处于待评价状态时，学生可在详情页点击评价按钮，前端向 `/api/student/repair-orders/{id}/rating` 接口提交评分和评价内容。后端保存评价记录，并将工单状态从 `pending_rating` 更新为 `completed`，同时插入流转记录，最终完成业务闭环。

## 5.4 宿管端功能实现

宿管端与管理员端、维修人员端共用 `admin-web` 前端应用，通过当前登录用户的角色决定页面功能展示范围。宿管角色登录后，系统显示工单管理、宿舍管理和公告管理等模块。

### 5.4.1 工单审核与分配实现

宿管进入工单管理页后，可查看工单列表并根据状态和关键词进行筛选。点击详情可以查看工单编号、学生姓名、维修人员、故障描述、现场图片和流转记录。对于状态为待审核的工单，宿管可以执行分配或驳回操作。

分配操作会调用 `/api/dorm-admin/repair-orders/{id}/assign` 接口，将维修人员编号和备注传给后端。后端更新工单状态为 `pending_accept`，写入分配时间和维修人员编号，同时在 `repair_flow` 中写入“宿管分配维修人员”的流转记录。驳回操作会调用驳回接口，后端将工单状态更新为 `rejected` 并记录驳回原因。

### 5.4.2 楼栋与宿舍管理实现

宿管端提供楼栋管理和宿舍管理功能，可新增楼栋、录入楼栋编码和楼层数量，也可新增宿舍并关联楼栋、宿舍号和容量。这些数据保存在 `dorm_building` 和 `dorm_room` 表中，为学生报修和统计分析提供基础数据支撑。

### 5.4.3 学生住宿信息维护实现

系统后端还提供学生住宿信息查询与调整接口，用于修改学生与宿舍之间的绑定关系。虽然当前前端页面未突出展示完整的学生住宿信息管理页，但相关接口与数据结构已具备，体现了系统对宿舍基础数据维护能力的预留。

### 5.4.4 公告管理实现

宿管可以在公告管理页录入公告标题和内容，并发布到系统中。公告保存至 `announcement` 表后，学生端即可同步查看。公告模块承担了系统公共信息通知的作用，是学生端首页的重要数据来源。

## 5.5 维修人员端功能实现

维修人员也通过 PC 管理端登录系统。当角色识别为 `repairer` 时，界面自动切换到“我的工单”页签，避免显示无关的宿管和管理员功能。

### 5.5.1 我的工单与接单实现

维修人员登录后可查看分配给自己的工单列表，列表中展示学生姓名、报修类型、当前状态和标题等信息。对于待接单工单，维修人员点击接单按钮后，前端向 `/api/repairer/repair-orders/{id}/accept` 发送请求，后端将工单状态更新为 `processing`，并写入“维修人员接单”的流转记录。

### 5.5.2 维修反馈实现

当工单处于处理中状态时，维修人员可打开维修结果填写窗口，输入维修结果描述、所用材料和完成时间，同时通过上传接口上传最多三张维修结果图片。提交后，系统将反馈信息写入 `repair_feedback` 表，将图片路径写入 `repair_order_image` 表，并把工单状态更新为 `pending_rating`。这一过程不仅完成了工单处理结果的记录，也为后续学生评价提供了可视化依据。

### 5.5.3 个人统计实现

后端为维修人员提供统计接口，可计算该维修人员处理工单总数、已完成数量和相关比例。虽然当前管理端页面主要展示工单列表和处理操作，但后端统计接口已经具备，体现了系统对维修人员绩效统计场景的支持。

## 5.6 管理员端功能实现

管理员登录管理端后可以访问系统的全部核心管理功能，其中重点包括统计概览、用户管理和日志查看。

### 5.6.1 统计概览实现

管理员主页通过调用统计概览接口获取总工单数、待审核工单数、处理中工单数、已完成工单数和平均评分等数据，并以统计卡片形式展示。与此同时，系统通过调用报修类型统计接口和楼栋热度统计接口，将结果交给 ECharts 绘制饼图和柱状图，实现管理数据的可视化展示。

### 5.6.2 用户管理实现

用户管理功能支持查看用户列表、新增用户、修改用户状态和重置密码。管理员可新增宿管、维修人员或管理员账号，系统默认密码为 `123456`。该模块提升了系统用户维护的便利性，也保证演示过程中能够快速生成测试账号。

### 5.6.3 日志与字典查看实现

系统通过 `sys_log` 表记录初始化、报修提交、工单分配、维修反馈和评价等关键操作，管理员可以在日志模块中查看这些记录。系统还提供字典列表接口，用于查看基础字典配置，例如报修优先级和公告类型等信息。虽然字典管理在当前前端中未进行复杂维护，但后端接口已经为后续扩展保留空间。

## 5.7 文件上传与静态资源访问实现

在宿舍报修场景下，图片能够有效增强故障描述和维修结果表达，因此文件上传是系统的重要辅助能力。本系统通过 `CommonController` 提供 `/api/common/upload` 接口，前端以 `multipart/form-data` 形式上传文件，后端接收后保存到本地目录，并返回可访问路径。

为使上传图片能够在前端直接查看，系统在 `WebConfig` 中将 `uploads` 目录映射为 `/uploads/**` 静态资源路径。这样，学生端和管理端在工单详情中都可以通过拼接 `http://localhost:8080` 与文件路径的方式进行图片展示和预览。

这种“数据库存路径、文件保存在本地目录”的方式实现简单、演示效果直观、依赖少，十分适合当前毕业设计项目。但从工程扩展角度看，若未来需要支持海量图片、权限化访问或云部署，还可进一步升级为对象存储方案。

## 6.4 测试结果分析

从测试结果来看，本系统已基本完成高校宿舍报修管理的核心业务需求。首先，系统实现了清晰的角色分工和流程闭环，使学生、宿管、维修人员和管理员在统一平台上围绕报修工单开展协同处理。其次，系统使用状态流转记录和日志记录增强了业务过程的可追踪性，为后续问题定位和管理分析提供了基础。再次，学生端和管理端的终端划分符合实际场景，移动端便于学生快速提交与查询，PC 端则便于后台管理和统计展示。

当然，测试中也能看出系统仍存在一定不足。当前前端页面集中在单页中完成大部分功能，组件拆分程度有限；管理员端虽然具备统计图表和用户管理能力，但日志筛选、字典维护等功能还较为基础；维修人员统计接口虽然已具备，但前端展示仍有增强空间；密码当前采用明文存储，尚未引入更完善的加密机制；部分列表查询未实现分页，对大规模数据场景的支持仍有待加强。

总体而言，本系统在毕业设计项目的范围内实现了较完整的功能闭环和较清晰的系统结构，达到了预期的设计与实现目标。

# 7 结论与展望

## 7.1 结论

本文围绕高校宿舍报修场景中的实际业务需求，设计并实现了一套基于 Spring Boot 与 Vue 的高校宿舍报修管理系统。系统采用前后端分离架构，以 SQLite 作为数据存储方案，按照学生、宿管人员、维修人员和管理员四类角色划分功能边界，完成了学生在线报修、宿管审核分配、维修人员接单反馈、学生评价以及管理员统计分析等核心业务模块的开发。

在研究与实现过程中，本文完成了系统需求分析、架构设计、数据库设计、接口设计、权限控制设计和详细实现分析，并结合实际运行结果开展了功能测试。测试结果表明，本系统能够实现宿舍报修流程的闭环管理，具备较好的可用性和演示性。相较于传统线下报修方式，系统在信息记录规范化、流程可追踪、维修进度透明化和数据统计直观化方面具有明显优势。

本项目作为本科毕业设计，在控制开发复杂度的前提下实现了较为完整的业务流程，既体现了软件工程、数据库原理、Web 开发技术和系统分析设计等课程知识的综合应用，也为高校宿舍后勤维修的信息化管理提供了一个可参考的原型方案。

## 7.2 展望

尽管系统已经实现了主要功能，但仍然存在一定提升空间，未来可以从以下几个方向继续完善。

第一，可引入更加完善的安全机制。例如对用户密码进行哈希加密存储，增加登录失败限制、验证码或操作审计等能力，提高系统安全性。

第二，可增强工单调度能力。当前宿管主要通过人工选择维修人员完成分配，后续可以根据维修类型、人员空闲情况或历史处理效率，引入自动派单机制。

第三，可完善权限模型。当前系统采用基于角色字符串的简单权限控制，未来可进一步扩展为细粒度 RBAC 权限体系，实现菜单级、接口级甚至按钮级权限管理。

第四，可完善移动端体验。学生端目前已能满足报修和查询需求，后续还可增加消息提醒、历史公告列表、个人资料维护优化和更丰富的工单筛选能力。

第五，可提升系统工程能力。例如增加分页查询、条件组合搜索、异常重试、单元测试与集成测试覆盖、容器化部署和云端文件存储等，使系统更接近实际生产环境要求。

综上所述，本系统已具备较好的基础功能和扩展潜力，在后续工作中可继续向安全性、智能化和工程化方向演进。

### 附录 A 主要数据表说明

1. `user`：系统用户表，保存账号、密码、姓名、角色、状态。  
2. `student_profile`：学生档案表，保存学号、学院、专业、班级、楼栋、宿舍等信息。  
3. `dorm_building`：楼栋表，保存楼栋名称和楼栋编码。  
4. `dorm_room`：宿舍表，保存楼栋关联、宿舍号和容量。  
5. `repair_order`：工单核心表，保存报修主题、描述、状态、关联学生和维修人员。  
6. `repair_order_image`：工单图片表，保存故障图片和维修结果图片。  
7. `repair_flow`：工单流转表，保存状态变化记录。  
8. `repair_feedback`：维修结果表，保存维修描述、材料和完成时间。  
9. `repair_rating`：评价表，保存评分与评价内容。  
10. `announcement`：公告表，保存公告标题和内容。  
11. `sys_log`：系统日志表，保存关键业务操作。  
12. `sys_dict`：基础字典表，保存通用配置项。
