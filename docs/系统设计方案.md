# 高校宿舍报修管理系统设计方案

## 1. 项目定位与开发方案

本项目定位为小型单体毕业设计项目，目标是功能闭环完整、实现难度可控、便于演示和答辩。系统采用前后端分离架构，分为 `PC 管理端 + 移动端 Web 学生端`。

技术栈确定如下：

- 后端：`Java 17 + Spring Boot`
- 数据库：`SQLite`
- 持久层：`MyBatis`
- PC 管理端：`Vue 3 + Vite + Element Plus`
- 移动端学生端：`Vue 3 + Vite + Vant`
- 图表：`ECharts`
- 文件上传：本地文件存储
- 接口风格：`RESTful API`
- 认证方式：账号密码登录 + 基于角色的简单权限控制

采用这套方案的原因如下：

- 满足任务书硬性要求：`Spring Boot + Vue`
- `SQLite` 无需单独安装数据库服务，部署简单，适合小项目
- 单体架构最适合当前项目规模
- PC 端适合宿管、维修、管理员进行审核、管理和统计
- 移动 Web 适合学生在手机浏览器提交报修和查看进度

## 2. 系统角色与终端划分

系统共四类角色：

- 学生：使用移动端 Web
- 宿管人员：使用 PC 管理端
- 维修人员：使用 PC 管理端
- 系统管理员：使用 PC 管理端

角色职责如下：

- 学生负责注册登录、提交报修、查看进度、评价反馈、查看公告
- 宿管负责审核报修、分配维修人员、管理宿舍信息、跟踪维修进度、发布公告
- 维修人员负责接单、维修处理、提交维修结果、查看个人统计
- 管理员负责用户管理、角色权限、统计分析、日志查看、基础配置

## 3. 最终系统流程图说明

### 3.1 学生报修流程

1. 学生登录移动端 Web。
2. 学生进入报修页面，填写宿舍号、故障类型、故障描述、期望维修时间，并上传现场图片。
3. 系统生成报修单，状态为 `待审核`。
4. 宿管在 PC 管理端查看待审核工单。
5. 宿管审核通过后，分配维修人员，工单状态变为 `待接单`。
6. 若审核不通过，则填写驳回原因，工单状态变为 `已驳回`。
7. 学生可在移动端查看审核结果和工单状态。

### 3.2 维修处理流程

1. 维修人员登录 PC 管理端。
2. 维修人员查看分配给自己的待处理工单。
3. 维修人员确认接单后，工单状态变为 `处理中`。
4. 维修完成后，填写维修结果、所用材料、维修说明，并上传维修后照片。
5. 系统将工单状态变为 `待评价`。
6. 学生查看维修结果后进行评分与反馈。
7. 学生评价完成后，工单状态最终为 `已完成`。

### 3.3 宿舍信息管理流程

1. 宿管维护楼栋、宿舍、入住学生、宿舍设施等基础信息。
2. 学生注册后可与宿舍信息绑定。
3. 报修时系统优先自动带出宿舍信息，减少重复填写。
4. 宿舍信息支持查询、修改和导出。

### 3.4 公告通知流程

1. 宿管或管理员在 PC 管理端发布公告。
2. 公告包括标题、内容、发布时间、适用范围。
3. 学生在移动端首页查看公告内容。

### 3.5 系统管理流程

1. 管理员登录系统。
2. 管理员维护用户、角色、基础字典。
3. 系统记录关键操作日志，如登录、审核、分配、修改权限等。
4. 管理员查看统计图表和日志信息。

## 4. 工单状态设计

建议统一采用以下状态：

- `待审核`
- `已驳回`
- `待接单`
- `处理中`
- `待评价`
- `已完成`

状态流转如下：

- 学生提交后：`待审核`
- 宿管驳回：`已驳回`
- 宿管通过并分配：`待接单`
- 维修人员接单：`处理中`
- 维修完成提交结果：`待评价`
- 学生评价后：`已完成`

如果后续需要进一步简化，也可以将 `待评价` 合并进 `已完成`，但为了保证流程完整性，当前方案保留 `待评价` 状态。

## 6. 数据库表设计草案

以下为基于 `SQLite` 的简化版表设计草案。

### 6.1 `user` 用户表

字段建议：

- `id`：主键
- `username`：登录账号
- `password`：登录密码
- `real_name`：真实姓名
- `phone`：手机号
- `role`：角色，取值为 `student`、`dorm_admin`、`repairer`、`admin`
- `status`：账号状态
- `created_at`
- `updated_at`

说明：

- 当前项目规模较小，可直接在用户表中保存角色字段
- 如后续需要扩展，再拆分角色和权限表

### 6.2 `student_profile` 学生信息表

字段建议：

- `id`
- `user_id`
- `student_no`
- `gender`
- `college`
- `major`
- `class_name`
- `building_id`
- `room_id`
- `bed_no`
- `created_at`
- `updated_at`

### 6.3 `dorm_building` 楼栋表

字段建议：

- `id`
- `building_name`
- `building_code`
- `gender_type`
- `floor_count`
- `remark`

### 6.4 `dorm_room` 宿舍表

字段建议：

- `building_id`
- `room_no`

### 6.5 `repair_type` 报修类型表

字段建议：

- `room_id`
- `repair_type_id`

### 6.8 `repair_flow` 工单流转记录表

- `id`

## 8. 技术选型说明

论文中可按以下思路描述技术选型：

- 后端采用 `Spring Boot`，便于快速搭建单体应用，提高开发效率
- 前端采用 `Vue 3`，能够较好实现组件化开发与页面交互
- 数据库采用 `SQLite`，由于本系统规模较小、部署环境简单，使用轻量级数据库可降低安装维护成本
- PC 管理端采用 `Element Plus`，便于实现后台管理页面
- 学生端采用移动 Web 方案，能够满足手机访问需求，同时降低小程序开发与部署复杂度

## 9. 实现边界建议

首版建议实现以下内容：

- 学生端完整闭环：提交报修、查看进度、评价反馈
- 宿管端完整闭环：审核、分配、宿舍管理、公告发布
- 维修端完整闭环：接单、处理、反馈
- 管理员端基础功能：用户管理、日志管理、统计分析、基础配置

首版不建议纳入以下高级功能：

- 微信小程序
- 消息推送
- 自动派单
- 复杂 RBAC 按钮级权限
- Redis
- Docker
- 对象存储
- 多数据库兼容

该方案在保证功能完整性的同时，能够最大限度降低开发复杂度，适合作为毕业设计项目实施方案。
